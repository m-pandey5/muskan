Experience

  • Umbra Privacy: Backend Engineer (December 2025 – April 2026)

    Backend work on a privacy-oriented cross-chain stack: NEAR Intents for intent-based execution, a production-grade Solana indexer in Rust for on-chain data, and a Solana gasless relayer (g-relayer) service.

    • Private bridge (NEAR Intents): I implemented a complete private bridge on top of NEAR’s intent system, routing user and protocol flows so that assets can move from any supported chain to any supported chain. The design uses an ephemeral keypair and keys derived with HKDF so that, when bridging, there is no on-chain linkability between source and destination—settlement and contracts stay correct for solvers and policy, but the on-chain graph does not reveal a traceable public link across domains. I owned the backend for composing intents with bridge logic, solvers, and checks so metadata exposure stays within the threat model, while staying compatible with NEAR’s intent and solver ecosystem.
    • Solana data indexer (Rust, Diesel): built a Solana data indexer in Rust that ingests on-chain state and activity (RPC and streaming, transactions, instruction traces, program accounts) and stores it with the Diesel ORM. I implemented end-to-end backfilling of historical events (including the full backfill and recovery flows), with gap repair, reorg- / fork-aware handling, and checkpointed pipelines so the service can be restarted safely; downstream use includes bridge monitoring, replay, and idempotent re-processing.
    • g-relayer (Solana gasless relayer, Rust): implemented a REST API where the relayer signs as the fee payer and submits transactions, so end users can sign without holding SOL: the relayer covers network fees and can be reimbursed in another token via a payment instruction. The service exposes JSON endpoints (for example config and payer info, health, supported tokens, fee estimation, building partially built transactions, signing, and sign-and-submit), with optional features such as Prometheus metrics, Redis-backed rate limits, and a Jupiter-based pricing feature for token-denominated fees. Configuration (RPC URL, key material, server port, rate limits, validation rules such as allow-listed programs, caps on lamports and signature count, and allow-listed SPL “pay the relayer” tokens) is driven by a TOML file or environment. The end-to-end flow is client-driven for blockhashes: the relayer can return unsigned transactions with a placeholder blockhash, the user or client replaces it with a recent blockhash and applies partial signing, and the relayer then completes fee-payer signing and submission after a validation pass—keeping blockhash sourcing explicit on the user side rather than the relayer.
